## 0. 용어 빠른 풀이 (이 문서를 읽는 데 필요한 만큼만)

| 용어 | 한 줄 뜻 (이 프로젝트 맥락) |
|---|---|
| **Diffuser** | 과거 주행 데이터를 보고 "앞으로 2.56초(128 step)의 주행 계획(궤적)"을 통째로 그려내는 생성 모델. |
| **prior(프라이어)** | Diffuser가 학습한 **"주행 습관/밑그림"**. 데이터가 어떤 주행으로 가득하면 그쪽으로 그린다. |
| **value(밸류)** | 한 주행 계획이 **"얼마나 좋은지" 점수**를 매기는 채점관. 점수가 높은 쪽으로 계획을 살짝 밀어준다(=guidance). |
| **guidance / scale** | value가 prior의 밑그림을 더 좋은 쪽으로 미는 **세기**. scale=0이면 안 밀고 prior 그대로(=순수 흉내내기, BC). |
| **stitching(스티칭)** | 데이터에 없던 조합을 **이어붙여** 새 주행을 만드는 능력. 예: "느리지만 안전한 완주" + "빠른 한 구간"을 합쳐 "빠른 완주". offline RL이 behavior(원본 정책)를 **넘어서는** 핵심 메커니즘. |
| **closed-loop(닫힌 루프)** | 차가 **자기 행동의 결과를 다시 입력으로 받아** 다음 행동을 정하는 실제 주행. 작은 오차가 누적될 수 있음. |
| **covariate shift** | 닫힌 루프에서 **연습 때 본 적 없는 상황**에 빠지면 모델이 점점 헤매는 현상. |
| **K-step / MPC** | 그린 128 step 계획 중 **앞 K개만 실제로 쓰고** 다시 그리는 방식. 현재 평가는 K=1(매 step 새로 그림). |
| **baseline** | 우리가 넘어야 할 기준 기록 = cap-5 정책의 2랩 **107.16s**. |

> 큰 그림: **데이터로 Diffuser를 학습 → "주행 계획"을 그리게 함 → value로 더 빠른 쪽으로 유도 →
> baseline(107s)보다 빠른 2랩 주행.** 환경에서 추가 데이터를 더 모으지 않고(=offline) 해내는 게 과제.

### 1.1 데이터를 "재료"로 다시 보기 (Diffuser prior 관점)
- **완주 궤적 = 52개**(cap5_full 22 + cap10_full 30). 처음부터 끝까지 2랩을 안전하게 돈 "정답 주행".
- **추출 고속랩 = 249개**(cap15 132 + cap20 117). 충돌로 끝난 에피소드지만, **충돌 직전까지(=lap-1)는
  깨끗한 고속 랩**(18~20s). `log_lap_time_s>0` 첫 시점까지 잘라내면 충돌 프레임 없이 고속 주행만 남는다.
  → **재수집 0으로 얻는 공짜 고속 재료.** (정확히는 cap5/cap10 충돌분까지 더하면 추출 lap-1 = 278.)
- **충돌 궤적 = 715개**. 대부분 짧게 충돌로 끝남. → "무엇이 충돌로 가는가"를 **value에게 가르치는 재료**.

### 1.2 왜 충돌이 93%(715/767)나 되나 — 의도된 수집 전략
[[008-diffuser-plan-v4]]의 결정: **"crash-only" 수집.** 완주 데이터를 일부러 적게 모았다. 이유는
"완주만 잔뜩 넣으면 Diffuser가 그냥 그걸 **흉내**(BC)낼 뿐, **stitching으로 더 빨라지는 걸 증명 못 한다**".
즉 충돌 중심 데이터로도 안전 완주를 **복원**하고 거기서 더 빨라지면 진짜 offline RL의 가치 입증.
→ **이 전략이 P6에서 역풍을 맞았다(아래 §2).**

## 2. 무슨 문제가 생겼나

### 2.1 P6 1차 평가 = 완전 실패
학습은 정상 완료(diffusion loss 0.003, value 상관 corr **0.98**). 그런데 실제 주행 평가:
- **5 에피소드 / 5 충돌, 완주 0, baseline 107s 미달.** 랩 자체를 못 끝냄.
- **모든 설정에서 실패**: K∈{1,2,3,5,7,10,30}, scale∈{0,0.1,0.3,1.0}, value 32개 중 best 선택 — 전부 충돌.

체계적 진단(013 §2)으로 **모델·normalizer·관측(obs)·파이프라인은 전부 정상**임을 배제했다(알려진 학습
obs를 넣으면 데이터와 거의 똑같은 action을 생성: step0 데이터2.5↔생성2.6 등). 그럼 무엇이 문제인가 →
**데이터 자체와 그 학습 방식**에 두 가지 구조적 문제가 있다. 014가 이를 정량으로 못박았다.

### 2.2 문제 ① prior 오염 — "디퓨저가 충돌하는 습관을 배웠다"
- **쉬운 설명**: 데이터의 93%가 "짧게 달리다 충돌"이다. Diffuser는 "전형적인 주행 = 곧 충돌로 끝남"을
  습관으로 익혔다. 그래서 그려내는 주행 계획이 **충돌하기 쉬운(crash-prone)** 계획이다.
- **닫힌 루프에서 벌어진 일**(013 §1): 매 step 새 계획을 그리는데(K=1), 정상 가속하다 **이유 없이 감속 →
  과조향 → 스핀 → 후진**(평균 명령 −4.0 m/s = 뒤로 감). 실제 화면으로 "제자리 주행" 확인.
- **→ 이게 계획의 출발점**: "충돌 궤적을 prior에서 빼면 이 후진/스핀은 사라진다." 014도 **타당**하다고 인정
  (§5-a). **단, 이건 안전을 살릴 뿐 더 빨라지게 하진 않는다.**

### 2.3 문제 ② D3 — "채점관(value)이 충돌을 더 좋아한다" (014가 정량 확증)
- **쉬운 설명**: value는 "빠를수록 고득점"을 배웠는데, **충돌 벌점이 너무 약해서 사라진다**. 그래서
  "충돌로 끝나는 고속 주행"을 "안전한 저속 완주"보다 **더 좋다고 채점**한다.
- **숫자로(014 §4, 할인 γ=0.99·128 step)**:

  | 주행 종류 | value가 매기는 점수(할인 return) |
  |---|---|
  | 고속이지만 충돌로 가는 주행 | **32.86** |
  | 안전한 저속 완주 | **8.12** |

  분해하면: 128 step 동안 쌓이는 속도 보상 **+23.25** ≫ 충돌 벌점 **−10 × 0.99¹²⁸ = −2.76**(할인으로
  거의 소멸). → **value는 충돌 쪽을 4배 선호.** value로 계획을 밀면(scale↑) **충돌 쪽으로 민다.**
- **왜 중요한가**: 우리 계획의 "진짜 개선" 엔진이 바로 value guidance다. **그 엔진이 반대 방향을 가리킨다.**
  현재 value(corr 0.98)는 "이 잘못된 점수표에 0.98로 충실히 맞춘" = **충실하게 틀린** 모델.

### 2.4 014의 추가 발견 — 고속 재료(249개)에도 함정이 있다
- **고속랩은 "지속가능한 빠름"이 아니라 "충돌 직전의 한계주행"이다**(014 §3). 추출한 lap-1의 **말미 50
  step이 가장 빠르고**(cap20 13.6→15.8 m/s), 그 페이스로 lap-2에 들어가 충돌한다.
  - cap15 고속랩의 **27%가 lap-2 시작 즉시 충돌**, 절반 이상 버틴 건 14%뿐.
  - → 이 스타일을 prior가 흉내내면 **닫힌 루프에서 똑같이 lap-2 충돌을 재현**할 위험.
- **데이터 믹스가 고속으로 73% 편중**(014 §6): 주행 prior 후보(약 340) 중 **249개(73%)가 한계-고속**이고,
  baseline 초과의 핵심인 cap10 완주는 **30개뿐**. → "300개니까 6배라 좋다"는 **에피소드 수**와 **닫힌 루프
  안정성**을 혼동한 것. 그냥 다 넣으면 prior가 한계페이스로 쏠린다.

### 2.5 정리 — 013의 원래 계획이 흔들린 두 지점

| 013의 암묵 가정 | 1차 소스 반박(014) | 결론 |
|---|---|---|
| "고속랩 249개 넣으면 빠른 prior" | 그건 충돌-직전 한계주행, 73% 편중 → lap-2 충돌 재현 위험 | 그냥 넣으면 위험 |
| "value 재사용(corr 0.98)으로 충돌 밖에서 속도 유도" | D3: value는 충돌 고속을 4배 선호 → guidance가 충돌로 민다 | value를 **고쳐야** 함 |

**통과한 부분**(014도 인정): 데이터 숫자(249·52)는 사실, 양(≈450k transition)은 병목 아님,
normalizer 통일 처방은 정확. → 문제는 **양이 아니라 "구성과 학습 방식"**이다.

## 3. 그래서 이렇게 진행하겠다 — 단계 게이트 (A → B)

### 3.1 큰 그림: 왜 한 번에 안 하고 단계로 나누나
013은 "완주+고속랩 300개를 prior에, 충돌은 value에" 한 번에 넣으려 했다. 문제는 **변수가 두 개 동시에
바뀐다**는 것: ①고속 재료 ②충돌-선호 value. 그러면 또 실패해도 **둘 중 뭐가 원인인지 모른다.**
→ **변수를 하나씩 분리해 게이트로 검증한다.** 안전을 먼저 확보(A), 그 위에서만 개선을 시도(B).

### 3.2 Track A — "깨끗한 prior면 안전 완주가 되는가"를 단독 검증
**목적**: 고속 재료·value 변수를 **빼고**, "충돌 궤적을 prior에서 빼면 후진/스핀이 사라지고 완주가
나온다"만 깨끗이 확인. 여기서 완주가 나오면 **baseline 107s 초과(문자적 성공)** 확보.

| 단계 | 할 일 | 구현 지점(1차 소스) | 비용 |
|---|---|---|---|
| A1 | 로더에 모드 플래그 `F1TENTH_MODE = complete \| driving \| all` 추가. **complete = 완주 52개만**(npz `is_terminal` 전부 False인 ep). ★ normalizer는 어떤 모드든 **전체(all) 통계로 통일** fit | `datasets/f1tenth.py` `f1tenth_sequence_dataset`에 ep 필터 추가(line 107~) | 소 |
| A2 | diffusion 재학습(완주 prior). value는 안 씀 | `scripts/train.py --config config.f1tenth` (★`n_train_steps`는 10000 배수 — 아니면 0 epoch) | GPU ~6-8h (데이터 작아 더 빨리 saturate 가능) |
| A3 | 평가: **scale=0(순수 BC), K=1**, episodes≥5 | `scripts/plan_f1tenth.py`(V_MAX=20, RL_project venv, cwd=vendor/diffuser) | 분~시간 |

**게이트 판정 기준**:
- ✅ **통과** = 후진 degenerate 소멸(명령 속도 양수 안정) + 완주율 의미있게 >0 + 2랩 lap time **107s 초과**
  → **Track B 진입**.
- ⚠️ **부분**(앞으로는 가는데 완주 못 함) = prior 오염은 풀렸으나 **K=1 누적오차(compounding) 잔존**
  → B의 **warm-start를 앞당겨** 먼저 도입.
- ❌ **실패**(여전히 충돌·후진) = 진단 자체 재검토(obs/conditioning 구조 의심), B 보류.

### 3.3 Track B — "진짜 개선(<56s)"을 시도 (A 게이트 ✅ 후에만)

| 단계 | 할 일 | 핵심 주의 | 비용 |
|---|---|---|---|
| B1 | **D3 수정 value 재학습**: npz의 충돌 보상성분(`log_reward_collision`)을 가중해 충돌 벌점 −10 → 강화. ★ **재학습 전에 무비용 재계산**으로 "고속-충돌 점수 < 안전-저속 점수" **부호 역전** 확인 후 채택 | 벌점이 **과하면** value가 빠른 모드를 일괄 회피 → stretch도 죽음. **−50/−100 등 1~2 후보 비교**, 일방 −300 금지. diffusion은 재사용 | GPU value만 ~5-10h |
| B2 | **고속 lap-1 추가**(`driving` 모드=완주+추출 lap1) prior 재학습, **분리 로깅** | 한계주행이라(§2.4) 안전 prior와 **분리**해 "어느 재료가 닫힌 루프를 깨는지" 추적 | GPU ~6-8h |
| B3 | **warm-start 샘플링**(조건부) | A가 ⚠️였을 때만. 매 step 백지에서 그리지 말고 **직전 계획을 초기값/조건**으로 이어 그림. 코어 무변경(plan_f1tenth.py 글루) | 소 |
| B4 | 평가: **scale 스윕(0/0.1/0.3) × K 스윕**. 목표 **<56s** | value가 충돌로 밀면 scale 낮춰 균형 | 시간 |

### 3.4 게이트 흐름도
```
A1 로더 → A2 학습(완주 prior) → A3 평가(BC) → [게이트]
                                                 ├─ ✅ 통과 ─► B1(D3 검증·수정) ─► B2(고속랩) ─► B4(스윕)
                                                 ├─ ⚠️ 부분 ─► B3(warm-start) ─► A3 재평가 ─► (✅면) B1...
                                                 └─ ❌ 실패 ─► 진단 재개, B 보류
```

## 4. 핵심 원칙 · 비용 · 결정사항

### 4.1 이 노선을 지키는 4원칙
1. **변수 분리** — A에서 고속재료·value를 빼야 "prior 청소가 효과 있나"를 깨끗이 측정. 일괄투입 금지.
2. **무비용 게이트 우선** — B1의 부호-역전 재계산은 npz 읽기(비파괴)라, value 재학습(~10h) **투입 전에**
   효과를 미리 거른다(헛 재학습 방지).
3. **over-plan 금지** — warm-start(B3)는 A가 ⚠️일 때만. 깨끗한 prior로 K=1이 안정되면 도입 안 함.
4. **재수집 후순위** — 양은 병목 아님(014 §8). A·B가 커버리지 부족을 실증할 때만 cap10 완주/고속랩 타깃 수집.

### 4.2 정직한 기대치 (정성평가 기준)
| 층위 | 방법 | 가장 그럴듯한 결과 | 보고서 의미 |
|---|---|---|---|
| **안전** | Track A (완주 prior, BC) | 56~60s대 완주 = baseline 107s 초과 | 문자적 성공이나 **모방**(개선 아님) |
| **진짜 개선** | Track B (D3 수정 + 고속랩 + scale) | <56s 노림(불확실) | offline RL의 **stitching** 입증 = 핵심 기여 |
- BC만으론 behavior(cap-10, 56s)를 못 넘는다(사용자 지적·014 동의). **진짜 차별화는 B에 있다 — A는
  안전망이지 목적지가 아니다.** 어느 쪽이든 **§2의 D3·closed-loop 실패 분석**은 정성평가 산출물로 유효.

### 4.3 함정 체크리스트 (실행 시 1차 소스 재확인)
- **normalizer 통일 필수**: prior(주행-only)와 value(전체)가 다른 통계면 value guidance가 망가지는데,
  `check_compatibility`(serialization.py:62)는 **타입만 검사**해 조용히 통과 → 안 잡힘. → **전체 통계로
  통일**(014 §7, 선형 affine이라 주행-only 학습 이점은 희석 안 됨).
- **config 값**(현재): discount **0.99**(config:78), normed **True**(:80), horizon **128**, dim_mults
  (1,4,8), SafeLimitsNormalizer(:40), 평가 scale 기본 0.1(:123). B1에서 reward 재구성 시 value만 영향.
- **학습 step**: `n_steps_per_epoch=10000`이라 `--n_train_steps`는 **10000 배수**여야 학습됨.
- **실행 규약**: GPU는 `run_in_background`(foreground+CUDA=exit144), 프로세스 종료는 PID(pkill 자기매칭
  금지), 평가는 **RL_project venv**(diffuser venv는 env 빌드 불가), cwd=vendor/diffuser.
- **코어 무변경**: temporal/diffusion/helpers/trainer 손대지 않음(ValueFunction fork-patch는 기존 예외 1건).
